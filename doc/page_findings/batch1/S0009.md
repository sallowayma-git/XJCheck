recognition_strategy: 当前应继续走 `WireDiagramExtractor`，但要启用“重复行带 + 跨端子桥接”的子模式，而不是改走表格识别器。原因很明确：92 个 line_group 全是 horizontal，且有效对象主要由 CONNECT 导线定义；脚本已经能稳定抓到每一行的左右半段，只是把同一物理行拆成了两条 pair。对本页更合适的流程是先按 y 轴聚成标准行带，再把“左短线段 + 中间端子符号/数字 + 右长线段”合并为一条物理回路，最后再抽取左右语义。

number_matching_strategy: 数字匹配应主要依赖导线附近的独立 DIM 文本，而不是表格格元或块内属性。当前最稳定的数字源就是紧贴导线断点的中间端子号，例如 `229/230/301-330/521-531`；这些文本的 x 位置高度稳定，基本落在三列锚点 x≈90.6、215.6、340.6 上。块基本是符号块（如 `SYMB2_M_PWF165/191/231`），数字并不主要存在于块属性里，所以“导线 + 独立文本”应是主策略，“列锚点/行带约束”应是辅助策略。

2. 页底公共横线 `G0359`（y≈245）穿过多个数字 `231/232/331/332/532`，现有端点窗口策略把它整体丢弃，说明需要“多锚点总线”特判。
3. 顶部三条 y≈77.5 的长横线更像列框/汇线，不是有效端子配对对象，当前应显式降权或过滤。
4. 右列 y≈145、135 的两行不是标准 `BI xx + 开入xx` 模式，而是告警说明文本；如果脚本强依赖标准模板，会持续报“右侧缺失”。
5. 左侧 `1-21QDxx`、右侧 `BI xx/开入xx` 主要是语义标签，不应被当成数字对端；当前很多 “missing left/right candidate” 实际是语义映射没做完，不是 OCR 丢字。

script_change_suggestions:
1. 在 `WireDiagramExtractor` 中增加 split-row merge：若两条同 y、相邻 x 的 CONNECT 水平线之间只隔一个端子符号/数字窗口，则合并成一条物理回路，再生成一条 pair，而不是两条单边 pair。
2. 增加页型规则：命中 `BINARY INPUT`/`开入回路` 且存在 3 个稳定列锚点、10 单位等间距行带时，切到 `binary_input_grid` 策略。
3. 为 `1-21QDxx + 端子号 + BI xx/开入xx` 建立语义抽取器：左侧读设备/回路代号，中央读端子号，右侧读通道名，三类文本不要互相抢 numeric candidate。
4. 对底部 y≈245-265 的 `GD/DC` 区增加独立规则：识别公共电源/公共线，不按普通开入行配对。
5. 对 `Power loss alarm`、`Device alarm` 这类说明行增加非标准右侧语义模板，避免将其一律判成“右侧缺失”。
6. 把 `source_block_name` 为空、位于 DIM 层、且落在稳定列锚点附近的独立数字文本设为高优先级；同步降低块内数字和远离端点的数字权重。
