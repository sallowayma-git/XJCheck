# DWG 配线图跨页端子数字匹配校验软件开发任务书

版本：v0.1
目标阶段：MVP 可用版
部署形态：本地离线运行
核心输入：一个项目目录下的多张 DWG 图纸
核心输出：异常清单、定位证据、审计报告，以及供内部运行使用的 findings / 缓存状态

## 1. 项目背景与核心判断

本项目要解决的问题不是“完整理解电气原理图”，而是把一套规范 DWG 图纸中的“线—端点—数字—配对—跨页一致性”抽取出来，判断跨页引用或跨图连接是否存在数字配对不一致、缺失、重复或低置信度风险。

成熟电气 CAD 产品的做法也不是单纯依赖 OCR 或视觉大模型。AutoCAD Electrical 的 Drawing Audit 可以检测导线和线号相关问题、修正并生成报告，同时支持跳转到错误位置；Electrical Audit 可在项目级或当前图纸级显示导线、组件、端子重复等问题，并提供 Go To 和导出报告能力。AutoCAD Electrical 的 Signal Arrows 也采用 source / destination 同名代码关联跨图导线网络，并把源线号复制到目标网络。EPLAN 的 Message Management 则把逻辑检查结果集中为消息，支持在线/离线检查、项目级或页面级检查、消息分类、跳转定位和修正建议；EPLAN 的 interruption point cross-reference 专门用于表示跨页连接，并自动建立页面引用。以上说明，本项目的最佳路线应是“结构化 CAD 对象解析 + 项目级规则引擎 + 可追溯问题清单”，而不是以 OCR/VLM 为主链路。

MVP 的第一目标是：输入一套约 20–30 张规范 DWG 图纸，本地程序先识别图纸类型，再按图纸类型选择对应识别器，提取每页图中的数字关系、端到端配对或表格映射，并在整套图纸范围内校验同类数字配对是否一致。

MVP 的审计目标必须进一步明确为三类一致性：

- 端子号对端子号的一致性。
- 端子号对语义端的对应一致性。
- 表格映射与图内端子映射之间的一致性。

MVP 的边界必须收紧：不识别电器元件型号，不理解元件功能，不自动改图，不做完整 DWG 编辑器，不依赖云端，不以 OCR 为主路径，不把 VLM 作为判断依据。OCR/VLM 只作为后续 fallback，用于文字对象缺失、DWG 转换损坏、局部截图复核或低置信度解释。

## 2. 产品目标

本软件的产品目标是建立一个本地运行的 DWG 配线图审计工具。用户把一个项目文件夹导入软件，软件扫描其中所有 DWG，生成结构化 findings 运行态，再基于 findings 执行跨页端子数字匹配规则，最后输出异常清单、定位证据和可复核的审计报告。

核心使用者是电气图纸审核人员、二次回路图校审人员、设计助理或希望减少人工查线工作量的工程人员。系统的价值不是替代最终人工审图，而是把高风险位置、低置信度位置和跨页不一致位置集中暴露出来，减少人工在 20–30 张图纸之间反复翻查的成本。

MVP 成功标准如下：

MVP 页型范围如下：

- 必须纳入审计：二次原理图、元件接线图、端子图、表格型图。
- 必须识别但默认不进入配对审计：普通背板图。
- 背板图若主体是规则化插件/端子表格，则应升级为“背板表格型图”进入审计，不能因为文件名含“背板图”而一律跳过。
- 稳定跳过：封面、目录、布置图及其他明确非审计页。

第一，能够批量读取一个项目目录下的 DWG，并稳定生成每张图的页码、图名、有效绘图区、文本对象、线段对象、候选连接线、候选数字和候选配对。

第二，能够识别大部分水平连接线两端的数字，并输出每条线的左右端数字候选列表，而不是只保存一个最终答案。

第三，能够在整套图纸范围内建立数字配对索引，发现以下问题：同一数字在不同页对应了不同数字，某个数字只有单向出现没有反向匹配，某个数字出现多个竞争配对，某条线附近数字不唯一导致低置信度。

第四，报告必须能让人工复核：每个问题至少包含文件名、页码、线 ID、坐标、数字、冲突对象、置信度、规则编号、原因说明和候选证据。

第五，系统在不渲染完整 DWG 的情况下仍然可用。第一版前端可以只显示表格和坐标，后续再扩展局部截图、SVG 叠加或 DWG/PDF 预览。

## 3. 推荐技术路线

MVP 采用“DWG 转 DXF + Python 几何解析”的路线。优先使用 ODA File Converter 或 ezdxf 的 odafc 插件将 DWG 转为 DXF，再由 ezdxf 抽取 TEXT、MTEXT、ATTRIB、LINE、LWPOLYLINE、INSERT 等对象。ezdxf 官方支持调用本地 ODA File Converter 在 DWG / DXF 间转换，也支持通过 `odafc.readfile()` 把 DWG 转为临时 DXF 后作为 ezdxf 文档加载；同时 ezdxf 提供对 TEXT、MTEXT、MLEADER、DIMENSION、ACAD_TABLE、INSERT 等 DXF 实体内容的读取方法。

但系统不应继续把所有图纸都交给同一套“大脚本”统一识别。推荐路线应升级为“通用 CAD 抽取 + 页级分类器 + 规则路由 + 专用脚本库”的模式：

- 通用抽取层只负责把 CAD 实体忠实抽出来。
- 页级分类器负责判断一张图属于哪一类图，而不是直接做数字配对。
- 路由层负责把图送到对应的专用识别器。
- 专用识别器按图种分别处理，例如：
  - 普通回路图：以真实导线路径上的端点和线上的数字关系为主，不能退化成“近邻任意数字”
  - 元件接线图：允许方向感知、块展开、块内虚拟文字、标签和局部模板规则协同判断
  - 端子图：以行锁定、列角色和代号提取为主，不再套用普通回路图的端点近邻策略
  - 表格型图：优先识别行列、单元格和列间映射，不强行走导线端点逻辑
  - 背板图：允许只做分类与跳过，不强制进入配对审计

尤其对表格型图，系统必须允许独立脚本库支持，而不是继续套用导线识别器。若图中主体信息位于规则表格中，应先识别表格结构，再识别表格单元格中的数字关系。

普通回路图中也存在“元件分区型信号回路”子模式，不能只按全页近邻数字配对处理。对于类似 `16 高低压侧操作箱信号回路.dwg` 的页面，系统应先识别图中独立元件分区，再把元件分区上方名称与分区内局部数字组合成完整逻辑端。

元件分区型普通回路图的专用要求如下：

- 该页仍属于二次原理图 / 普通回路图，推荐保持 `WireDiagramExtractor` 路由，但应进入 `component_prefixed_signal_circuit` 子模式。
- 先按矩形边界、垂直分隔线、连接线簇和顶部标题位置识别独立元件分区。例如 `16 高低压侧操作箱信号回路.dwg` 中应识别两个分区：左侧 `1-2n`，右侧 `3-2n`。
- 每个元件分区上方的短文本作为分区前缀，只抽取类似 `1-2n`、`3-2n` 的名称；页面标题、说明文字和其他非分区顶部文本不得作为前缀。
- 分区内部或分区边界附近的局部数字，例如 `218`、`221`、`217`，不是独立跨页端子号；必须与所属分区前缀直接拼接成完整逻辑端，例如 `1-2n218`、`3-2n218`。
- 同一连接线外侧的端子文本，例如 `1-4YD1`、`1-4YD4`、`3-4YD1`，是跨页审计匹配端。
- 每条连接应输出“分区前缀 + 局部数字 -> 外侧端子”的高置信映射。例如 `1-2n + 218 -> 1-2n218`，再与左侧端子 `1-4YD1` 建立 `1-2n218 -> 1-4YD1`。
- 该子模式下，局部数字、分区前缀和外侧端子都必须保留坐标、线段和分区 bbox 证据，便于人工复核。
- `16 高低压侧操作箱信号回路.dwg` 已按实图核验：左侧分区顶部文本 `1-2n`，右侧分区顶部文本 `3-2n`；上方连接行内左侧外部端子 `1-4YD1` 与局部数字 `218` 位于同一连接带，应输出 `1-2n218 -> 1-4YD1`。旧式 `? -> 218`、`218 -> ?` 或把 `218` 裸用为跨页端子的结果均视为未命中该子模式。

普通回路图还存在“线中元件端口型”子模式。它不是全页元件分区，而是一个元件符号直接嵌在导线中间，元件两侧以局部端口号 `1`、`2` 标记端口，左右两侧分别接到外部端子或本页前缀端子。

线中元件端口型普通回路图的专用要求如下：

- 该页仍属于二次原理图 / 普通回路图，推荐保持 `WireDiagramExtractor` 路由，但应进入 `inline_component_port_circuit` 子模式。
- 先沿水平导线路径识别嵌在线中间的元件文本，例如 `3-2KLP1`、`1KLP1`。元件文本本身不是普通端子号，而是元件本体编号。
- 元件左右两侧或下方相邻的局部数字 `1`、`2` 是该元件的端口序号，不得作为普通端子裸参与 PairBuilder。
- 每个端口逻辑端应由“元件本体编号 + `-` + 端口序号”合成，例如 `3-2KLP1-1`、`3-2KLP1-2`、`1KLP1-1`、`1KLP1-2`。
- 端口左侧若连接到外部端子文本，例如 `3-2QD2` 或 `1QD2`，则输出 `元件本体编号-1 -> 左侧端子` 的结构化映射。
- 端口右侧若连接到局部数字，例如 `116`，应先结合本页/本区域前缀合成为完整逻辑端，例如 `3-2n116` 或 `1n116`，再输出 `元件本体编号-2 -> 合成端子` 的结构化映射。
- `10 低后备保护及信号回路.dwg` 已按实图核验：同一连接带内存在 `3-2QD2`、`3-2KLP1`、`116`，端口号 `1`、`2` 位于 `3-2KLP1` 附近，应输出 `3-2KLP1-1 -> 3-2QD2` 与 `3-2KLP1-2 -> 3-2n116`。
- `08 差动保护及信号回路.dwg` 已按实图核验：同构结构为 `1QD2`、`1KLP1`、`116`，应输出 `1KLP1-1 -> 1QD2` 与 `1KLP1-2 -> 1n116`。
- 旧式 `1 -> 116`、`2 -> 116`、`? -> 116` 或把 `116` 裸用为跨页端子的结果均视为未命中该子模式。

普通回路图还存在“测控开入矩阵型”子模式。它在文件名和页级分类上仍是二次原理图 / 普通回路图，但局部结构表现为多列规则行带：每列顶部给出裸数字端的前缀，例如 `1-21n`；每行左侧或同列左部给出完整端子，例如 `1-21QD12`；同一行右侧给出裸数字，例如 `127`。这类图不能只按普通导线端点近邻搜索，否则会出现 `? -> 127` 这类左侧缺失伪问题。

测控开入矩阵型普通回路图的专用要求如下：

- 该页仍保持 `WireDiagramExtractor` 路由，但应在抽取器内部进入 `input_matrix_wire_mapping` 子模式。
- 先识别列级前缀文本，例如顶部重复出现的 `1-21n`。该前缀只用于同一列裸数字端补全，不得作为独立 endpoint。
- 再按水平行带识别同一行内的完整端子文本和裸数字端。完整端子形如 `1-21QD12`、`1-21QD28`、`1-21QD44`；裸数字形如 `127`、`212`、`228`。
- 裸数字必须结合其所在列的前缀合成为完整逻辑端，例如 `1-21n127`、`1-21n212`、`1-21n228`。
- 每一行输出“完整端子 -> 前缀补全后的裸数字端”的结构化 `wire_component_mapping`。例如 `1-21QD12 -> 1-21n127`、`1-21QD28 -> 1-21n212`、`1-21QD44 -> 1-21n228`。
- 中文/英文功能说明，例如 `开入32/对象6分位`、`BI 32/Object 6 Opening position`，只进入 `semantic_channel` 或说明证据，不参与 endpoint 竞争。
- 该子模式的关键证据是列前缀、行带、同一行完整端子、同一行裸数字、裸数字所在列，而不是单条短线两端最近文本。若系统已经抽到 `1-21QD12` 和 `127`，却输出 `? -> 127`，应归类为子模式未命中或配对器未采用行列上下文。
- `08 测控1开入回路图1.dwg` 已按 phase60 抽取结果和人工确认核验：同一行附近存在 `1-21QD12` 与 `127`，顶部列前缀为 `1-21n`，应输出 `1-21QD12 -> 1-21n127`；同构行应输出 `1-21QD28 -> 1-21n212`、`1-21QD44 -> 1-21n228`。

元件接线图也必须允许独立脚本库支持，而不是继续把每条局部线段都当作普通回路导线处理。对于包含多个分离独立子元件的元件接线图，系统应先识别“子元件实例”，再在每个子元件内部建立“本体编号 + 输出口序号 -> 外部接线端”的结构化映射。

元件接线图的专用要求如下：

- 先按几何边界、块边界、圆形编号区、输出端口区和相邻间距，把页面拆成一个个分离独立的子元件实例。
- 每个子元件最上方圆形位置内的文本作为本体编号，例如 `1-40DK1`、`1-40DK2`。
- 子元件下方通常存在 `4输出` 或 `6输出` 等输出数量提示；该提示用于确定端口数量和端口序号范围。
- 对 `4输出` 子元件，应识别端口 `1..4`；对 `6输出` 子元件，应识别端口 `1..6`。若输出数量缺失，应保留为低置信候选，不得盲目补全。
- 每个端口的逻辑端应由“本体编号 + `-` + 端口序号”合成。例如本体编号 `1-40DK2` 的第 `1` 口应生成 `1-40DK2-1`，第 `2` 口应生成 `1-40DK2-2`。
- 每个端口附近或导出位置的外部接线文本，例如 `ZD17`、`ZD7`，应作为跨页审计匹配端。
- 同一子元件内，每个端口逻辑端与其外部接线端生成一条结构化映射，例如 `1-40DK2-1 -> ZD17`、`1-40DK2-2 -> ZD7`。
- 这类映射应作为 `ComponentDiagramExtractor` 的高置信结构化证据进入项目级审计，而不是被降级为普通“线端近邻数字”。
- 块内固定引脚号、局部端口序号和本体编号本身不能单独作为普通端子号参与 PairBuilder 排名；它们必须在子元件实例上下文中组合成完整逻辑端后再进入审计。

`21 元件接线图1.dwg` 中的 `4输出` / `6输出` 实例已按人工截图标注确认，开发时应按端口格子位置而不是普通线端近邻进行匹配。

`4输出` 元件的专用端口布局如下：

- 上方圆形内文本是本体编号，例如 `5DK`。
- 下方矩形被分成左右两列、上下两排，端口序号固定为：左上 `1`、左下 `2`、右上 `3`、右下 `4`。
- 每个端口对应同一格子外侧最近的接线端文本。外侧文本前可能带 `&`，`&` 只作为连接/引用标记保留在证据中，不进入规范化端子名。
- 人工标注样例中应输出：`5DK-1 -> ZD12`、`5DK-2 -> 5FD25`、`5DK-3 -> ZD4`、`5DK-4 -> 5FD1`。

`6输出` 元件的专用端口布局如下：

- 上方圆形内文本是本体编号，例如 `1-2ZKK`。
- 下方矩形被分成三列、上下两排，端口序号固定为：上排从左到右 `1`、`3`、`5`，下排从左到右 `2`、`4`、`6`。
- 每个端口对应同一列外侧的接线端文本：上排端口绑定上方外侧端，下排端口绑定下方外侧端。外侧文本前的 `&` 不进入规范化端子名。
- 人工标注样例中应输出：`1-2ZKK-1 -> 1-2UD1`、`1-2ZKK-2 -> 1-2n719`、`1-2ZKK-3 -> 1-2UD3`、`1-2ZKK-4 -> 1-2n720`、`1-2ZKK-5 -> 1-2UD5`、`1-2ZKK-6 -> 1-2n721`。
- `4输出` / `6输出` 的端口号只在所属子元件实例内有效，不能与其他相邻子元件共享，也不能作为普通 `ordinary_pair` 候选。

`23 元件接线图3.dwg` 属于元件接线图的新子类，可归入 `ComponentDiagramExtractor` 的 `strip_two_port_component` 子模式。它和 `21 元件接线图1.dwg` 的共同点是：都必须先识别上方圆形位置内的文本作为子元件本体编号，再用端口序号组合成本体端口；差异是该页下方元件呈长条形，长条两端各有圆形端口，圆内端口号通常为 `1`、`2`。

长条双端口元件接线图的专用要求如下：

- 先按块实例、长条形外框、两端圆形端口和上方圆形编号区识别一个子元件实例。
- 上方圆形编号区内文本作为本体编号，例如 `5KLP10`。
- 下方长条两端圆形内的 `1`、`2` 是端口序号，必须结合本体编号生成完整端口，例如 `5KLP10-1`、`5KLP10-2`。
- 端口附近的外部接线文本可能是同类元件端口，例如 `5KLP9-1`，也可能是本页前缀端子，例如 `5n112`；二者都应作为 `component_mapping` 的外部端。
- 端口方向按长条几何确定：上端圆形端口为 `1`，下端圆形端口为 `2`。端口 `1` 绑定长条上方或上端附近的外部接线文本；端口 `2` 绑定长条下方或下端附近的外部接线文本。
- 若某个端口外侧文本包含逗号分隔的多个外部端，例如 `5KLP3-1,5KLP2-1`，必须拆成多个独立 `component_mapping`，不得把整串文本当作一个端子名。
- `23 元件接线图3.dwg` 已按实图核验：同一子元件实例中，上方本体编号为 `5KLP10`；块 `FJL-25-2A_Mirror` 展开后存在圆形端口 `1`、`2`；端口 `1` 对应外部端 `5KLP9-1`，端口 `2` 对应外部端 `5n112`。应输出 `5KLP10-1 -> 5KLP9-1` 与 `5KLP10-2 -> 5n112`。
- 人工标注样例还确认：本体编号 `5KLP5` 的端口 `1` 连接上方外部端文本 `5KLP3-1,5KLP2-1`，应拆分输出 `5KLP5-1 -> 5KLP3-1` 与 `5KLP5-1 -> 5KLP2-1`；端口 `2` 连接下方外部端 `5n307`，应输出 `5KLP5-2 -> 5n307`。
- 旧式把圆内 `1`、`2` 当成普通端子，或把 `5KLP10` 本体编号直接与 `5n112` 裸配对，均视为未命中该子模式。

`21 元件接线图1.dwg` 与 `19 元件接线图1.dwg` 中还存在“小型端口盒元件”子模式，可归入 `ComponentDiagramExtractor` 的 `small_port_box_component` 子模式。该子模式不是普通导线配对，也不是说明性文字；它由上方圆形本体编号、下方矩形端口盒、端口盒内局部端口号、端口格外侧接线端文本共同组成。

小型端口盒元件的专用要求如下：

- 先识别上方圆形或近似圆形内的本体编号，例如 `AK`、`A'`、`KZKK`、`JR`。
- 再识别下方矩形端口盒。端口盒内的 `1`、`2`、`3`、`4` 是局部端口号，必须结合本体编号生成完整逻辑端，例如 `KZKK-1`、`JR-2`，不得作为普通端子裸参与 PairBuilder。
- 端口盒外侧贴近对应端口格的文本是外部接线端。外部文本前后的 `&` 只作为连接/引用符号保留在证据中，规范化 endpoint 时应去掉。
- 两端口纵向布局按上下格绑定：端口 `1` 绑定上方外侧文本，端口 `2` 绑定下方外侧文本。例如人工标注确认 `AK-1 -> JD1`、`AK-2 -> A'-1`，`A'-1 -> AK-2`、`A'-2 -> JD6`。
- 两端口横向布局按左右格绑定：端口 `1` 绑定左侧或同列外侧文本，端口 `2` 绑定右侧或同列外侧文本。例如人工标注确认 `JR-1 -> K-3`、`JR-2 -> K-4`。
- 四端口盒布局按左右两列、上下两排绑定：上排左格为 `1`、下排左格为 `2`、上排右格为 `3`、下排右格为 `4`。例如人工标注确认 `KZKK-1 -> JD8`、`KZKK-2 -> K-5`、`KZKK-3 -> JD3`、`KZKK-4 -> K-6`。
- 该子模式应输出 `component_mapping`，并参与跨页审计；不能降级为 `ordinary_pair` 的 `KZKK-4 -> ?`、`JR-1 -> ?`、`K-3 -> ?` 等半边缺失结果。
- 子模式路由应发生在 `ComponentDiagramExtractor` 内部：页级路由仍是元件接线图，但组件抽取器内部需要根据端口盒几何选择 `small_port_box_component`，不能把这类结构交给普通导线 PairBuilder 主路径。

`ComponentDiagramExtractor` 的 MVP 完成标准建议明确为：

- 能先把页面稳定识别为元件接线图，并保持 `ComponentDiagramExtractor` 路由，不被强网格、表格线或块密度误送进普通回路/表格识别器。
- 能在一页中识别多个独立子元件实例，并为每个实例输出 bbox、圆形本体编号、输出数量或长条双端口结构、端口序号和外部接线端。
- 能输出每个端口的结构化映射记录，例如 `component_id=1-40DK2`、`port_no=1`、`logical_endpoint=1-40DK2-1`、`external_endpoint=ZD17`。
- 能把这些结构化映射转成项目级可审计关系，并保留源文本、坐标、子元件实例 ID、端口位置和置信度。
- 当子元件结构稳定时，组件映射优先级高于普通端点近邻配对。

几何处理使用 Shapely。Shapely 的 STRtree 是 query-only R-tree 空间索引，可以基于几何 bounding box 快速查询邻近对象；这适合本项目中“找某条线端点附近的数字”“找共线碎线”“找候选连接线附近文本”等场景。项目级关系分析使用 NetworkX，因为它适合构造、操作和分析复杂网络，本项目可把数字、配对、页面、线段建成图结构，用于发现一对多、多对一、孤立节点、冲突簇和连通分量异常。

若 DWG 转 DXF 保真度不足，则第二阶段升级为 ODA Drawings SDK 或 Autodesk RealDWG。RealDWG 是 Autodesk 提供的 DWG/DXF 读写 SDK，支持 C++ 和 .NET 读取、写入 AutoCAD 创建的 DXF/DWG；ODA Drawings SDK 面向 DWG/DGN，提供访问、查看、创建、编辑、保存和导出能力，并支持 DWG/DGN/PDF/SVG/raster 等格式转换。MVP 不应一开始接入这些重 SDK，但架构上要保留 Reader Adapter，以便后续替换解析后端。

推荐依赖如下：

运行环境：Windows 11 优先，Python 3.11 或 3.12。
CAD 转换：ODA File Converter，本地安装。
DXF 解析：ezdxf。
几何索引：Shapely。
关系图分析：NetworkX。
数据处理：pandas、pyarrow、openpyxl。
后端服务：FastAPI。
本地任务执行：Typer 或 Click CLI。
前端：Streamlit MVP，或 FastAPI + React/Tauri 作为后续版本。
配置：YAML。
日志：structlog 或 loguru。
测试：pytest、hypothesis、golden file regression。
报告：Markdown、Excel、JSON、HTML。

## 4. 总体架构

系统采用分层架构，不把读图、规则和前端混在一起。

第一层是 Ingest Layer，负责项目导入、DWG 文件扫描、文件名排序、页码初步推断、DWG 转 DXF、转换日志和失败重试。

第二层是 CAD Extraction Layer，负责从 DXF 中抽取原始对象，包括文本、线段、折线、块、块属性、图层、布局、坐标和 handle。

第三层是 Layout Understanding Layer，负责从图纸结构中推断页码、标题栏、图框、有效绘图区、主要线区、非审计区。它不理解电气语义，只理解版面和坐标。

第四层是 Page Classification Layer，负责基于标题、文本密度、横竖线分布、规则网格、块密度、表格特征和历史规则，把每页图分类为若干图种。第一版分类器优先采用“规则 + 几何统计特征”的方式，而不是直接依赖视觉大模型。最少应覆盖：

- 二次原理图
- 元件接线图
- 端子图
- 表格型图
- 背板图
- 布置图
- 封面 / 目录 / 非审计页

页级分类结果除 `page_type` 外，还应明确 `audit_disposition`，至少包括：

- `audit_required`
- `classify_only`
- `skip_stable`

第五层是 Page Router Layer，负责把不同图种分发给不同识别器，而不是所有图都走同一条 PairBuilder。推荐至少具备以下路由目标：

- `WireDiagramExtractor`
- `ComponentDiagramExtractor`
- `TerminalDiagramExtractor`
- `TableExtractor`
- `LayoutOnlyExtractor`
- `SkipExtractor`

第六层是 Geometry Normalization Layer，负责把原始线段合并为候选连接线，把碎线、共线线段、近似水平线、线端点和候选端点区域标准化。该层只应服务于需要导线几何的图种，不应强制要求表格型图先构造导线。

第七层是 Pair Building Layer，负责在线的左右端点附近寻找数字文本候选，生成 PairCandidate 和 Pair。它必须保留所有候选及分数，不能只保存 winner。

第八层是 Rule Engine Layer，负责基于 Pair 和 PairCandidate 在项目级运行一致性规则，输出 Issue。它同时应允许接收来自表格识别器的高置信映射结果，并把它们作为独立信源参与校验。

第九层是 Confidence & Explanation Layer，负责为 Pair 和 Issue 生成置信度、原因说明和证据链。

第十层是 Report & UI Layer，负责消费内部 findings 状态，生成前端问题展示以及按需导出的审计报告、Excel、HTML。

表格型图的专用要求如下：

- 若一张图主体表现为规则表格，则优先识别表格骨架、行、列、单元格和跨列对应关系。
- 第一版重点支持“表头型三列表格”：
  - 表头单元格承载语义端前缀，例如 `1-21QD`。
  - 中间列承载从 `1` 开始向下递增的行号。
  - 左右两侧单元格承载需要跨页审计匹配的接线端或端子文本，例如 `1-21n552`、`1-21n553`。
- 对表头型三列表格，应把“表头前缀 + 行号”合成为逻辑语义端；例如表头 `1-21QD`、行号 `1` 应生成 `1-21QD1`，行号 `2` 应生成 `1-21QD2`。
- 同一行内，每个左右两侧接线端都应与该行逻辑语义端生成一条高置信 `table_mapping`；例如 `1-21QD1 -> 1-21n552`。
- 中间列行号本身不是普通端子号，不能被单独降级为普通端点近邻候选。
- 这类高置信表格映射可用于反查其他图中的端子配对、跨页配对、端子到语义端一致性和异常项。
- 若真实样本中的表格页不足，则允许用隔离的合成/变异样本补足 `TableExtractor` 验证，但必须与真实项目样本区分保存和评估。

`TableExtractor` 的 MVP 完成标准建议明确为：

- 能先把页面稳定识别为表格主导页，而不是误送进普通导线识别器。
- 至少稳定支持表头型三列表格，并恢复“表头前缀 + 行顺序 + 列角色 + 单元格文本”。
- 能把每一行输出为结构化映射记录，而不是只输出散落文本。
- 能明确输出每行合成的逻辑语义端，例如 `1-21QD1`、`1-21QD2`。
- 能把同一行左右两侧的 `n###` 接线端分别输出为 `table_mapping`，并保留表头、行号、源文本、单元格坐标作为证据。
- 当表格结构稳定时，表格映射的优先级高于端点近邻配对。
- 若真实样本存在表格页，则真实样本必须过；若真实样本不足，则必须额外提供隔离合成/变异表格样本做命中验证。

背板表格型图的专用要求如下：

- 普通背板图可以默认 `classify_only`，但若背板页主体表现为插件/端子规则表格，则应路由到 `TableExtractor` 或背板表格专用子模式进入审计。
- 典型样本是 `20 非电量保护背板图.dwg`。该页当前标题/文件名属于背板图，但 DXF 中的真实表头和行号位于块 `WBH-814E-E1SA-101` 的 `INSERT.virtual_entities()` 内；因此背板表格型图必须允许展开块内虚拟文本，不能只依赖普通 modelspace 文本。
- 表头文本可能带括号或中文说明，例如 `NKR308A(非电量选配)`。识别时应把中文说明和括号内容作为说明性证据旁路，规范化表头前缀为 `NKR308A`，不得把中文说明参与匹配。
- 表头下方左右两列行号，例如 `01`、`02`、`03`，应规范化为端口序号 `1`、`2`、`3`。
- 同一行外侧的接线端文本，例如 `5FD15`、`5FD16`，应作为跨页审计匹配端。
- 背板表格型图应输出“表头前缀 + `-` + 行号 -> 外侧接线端”的高置信映射。例如解析到表头 `NKR308A(非电量选配)`、行号 `01`、外侧接线端 `5FD15` 时，应输出 `NKR308A-1 -> 5FD15`。
- 如果用户或样本中出现 `NKR380A` 等同类表头，也应按同一规则处理：先剥离中文说明，再使用规范化表头和行号合成逻辑端。
- 图内中文功能说明，例如“调压重瓦斯开入”“调压压力释放开入”，只用于确认表格行语义或辅助人工复核，不能牵引脚本把匹配目标转向中文说明。

`18 高低后备保护背板图.dwg` 属于背板表格型图中的“背板插件端子表”子模式。该模式不是左右端子互相配对，也不是单纯的三列表格；它以插件表头和左右序号列为主信源。

背板插件端子表的专用要求如下：

- 顶部表头是插件/模块名称，人工标注确认样例表头为 `NCZ343A`。若表头周围存在括号、乱码、中文说明或装置说明，应保留为证据，但规范化前缀使用 `NCZ343A`。
- 表格左右各有一列端口序号，序号值必须按真实表格文本读取，不能依赖奇偶号、连续号或固定列数硬编码。当前样例表现为左侧 `01,03,05...31`、右侧 `02,04,06...32`，这只是样例现象，不是算法前提。
- 每个外侧端子应按真实几何关系绑定到同侧、同行、同一端口格的序号；如果后续样本出现非奇偶分布、缺号、跨行或合并格，应以表格线、单元格 bbox、文本中心和最近同侧端子关系为准。不得把同一行左右外侧端子互相配成普通 pair。
- 中间中文功能说明，例如“合后/手跳信号”“手跳开入”等，只进入 `semantic_channel` 或说明证据，不参与端子匹配目标。
- 外侧文本前后的 `&` 只作为连接/引用标记保留在证据中，不进入规范化端子名。
- 人工标注样例中应输出 `NCZ343A-1 -> 1-4CD5`、`NCZ343A-2 -> 1-4QD17`、`NCZ343A-3 -> 1-4QD18`、`NCZ343A-4 -> 1-4QD13`、`NCZ343A-7 -> 1-4PD3`、`NCZ343A-8 -> 1-4PD7`、`NCZ343A-31 -> 1-4PD5`、`NCZ343A-32 -> 1-4CD4`。
- 行号规范化时应去除前导零，例如 `01 -> 1`、`02 -> 2`；但原始文本、所在侧、行 bbox 和对应外侧端坐标必须保留。

推荐模块依赖关系：

`Ingest -> Extract -> Layout -> PageClassifier -> PageRouter -> Normalize/PerTypeExtractor -> PairBuilder/TableBuilder -> RuleEngine -> ConfidenceEngine -> Report/UI`

禁止反向依赖。RuleEngine 不得直接读取 DWG/DXF；它只能读取 findings。UI 不得重新运行几何推理；它只能展示 findings 和 audit report。Agent 优化规则时，除非显式进入“重新读图阶段”，否则只能基于 findings 重算。

## 5. 本地 Agent 工作流

本地 Agent 的工作必须分成两个阶段：先读图生成 findings，再基于 findings 校验。

### 5.1 阶段 A：Read & Findings

输入：一个项目目录，包含多张 DWG。
输出：内部 findings 运行态数据。它可以存在于内存、SQLite、缓存目录或临时文件中，但默认不是最终交付物。

流程如下：

1. 扫描项目目录，识别 DWG 文件，按文件名中的页码、标题栏页码或用户配置排序。
2. 为每个 DWG 生成唯一 `sheet_id`，记录文件路径、文件名、原始大小、hash、转换状态。
3. 调用 ODA File Converter 或 ezdxf odafc，将 DWG 转为 DXF。转换必须记录日志、版本、耗时、是否 audit。
4. 用 ezdxf 读取 DXF，提取 modelspace / paperspace。第一版优先处理 modelspace；如样本显示对象在 paperspace，则扩展 layout adapter。
5. 提取所有 TEXT、MTEXT、ATTRIB、INSERT、LINE、LWPOLYLINE、POLYLINE、ARC、CIRCLE 等对象。MVP 可只处理 TEXT、MTEXT、ATTRIB、LINE、LWPOLYLINE。
6. 计算每个文本对象的 bbox、插入点、旋转角、文字高度、图层、内容、normalized_text、是否纯数字、是否像页码、是否像端子数字。
7. 计算每条线的 start、end、length、angle、is_horizontal、layer、bbox。
8. 自动推断图框和标题栏。若无法稳定推断，则允许使用配置项固定有效审计区域。
9. 基于标题、文本分布、线分布、表格特征和块特征，对每页图先做分类，得到 `page_type`、`page_type_confidence` 和结构特征摘要。
10. 根据图纸类型把页面送到对应的识别器，而不是所有页面都走同一条几何配对链。
11. 对导线型页面，合并共线近邻线段，生成 `LineGroup`，并优先恢复被 inline 数字或标签切断的同一路径，再基于线端与线身关系查找文本候选。
12. 对普通回路图的结构化子模式，先识别普通导线路径之外的结构上下文，再建立映射。若命中元件分区型信号回路，先识别独立元件分区和顶部前缀，再把分区内局部数字合成为完整逻辑端，并与同一连接线外侧端子建立映射；例如 `1-2n + 218 -> 1-2n218`，再输出 `1-2n218 -> 1-4YD1`。若命中线中元件端口型回路，先识别嵌在线中的元件本体编号和端口 `1/2`，再输出 `3-2KLP1-1 -> 3-2QD2`、`3-2KLP1-2 -> 3-2n116` 这类端口映射。若命中测控开入矩阵型回路，先识别列前缀和水平行带，再把同一行裸数字补全为完整端子，例如 `1-21QD12 -> 1-21n127`。
13. 对元件接线图页面，先识别分离独立的子元件实例、圆形本体编号、输出数量或长条双端口结构和端口序号，再生成“本体编号-端口序号 -> 外部接线端”的组件映射；例如 `1-40DK2-1 -> ZD17`、`5KLP10-1 -> 5KLP9-1`、`5KLP10-2 -> 5n112`。
14. 对表格型页面，先识别表格骨架、表头、行列和单元格，再生成表格映射；第一版重点支持表头型三列表格，把“表头前缀 + 中间列行号”合成为逻辑语义端，并与同一行左右两侧的 `n###` 接线端分别生成高置信 `table_mapping`。
15. 对背板表格型页面，即使 `sheet_category` 是背板图，也要先展开必要的 `INSERT.virtual_entities()`，识别插件/端子表格中的表头、行号和外侧接线端；例如 `NKR308A(非电量选配)` + `01` + `5FD15` 应生成 `NKR308A-1 -> 5FD15`。
16. 为每个页面生成“页级 findings 记录”，至少说明：这张图是什么图、图内结构是什么、当前采用什么识别器、数字是如何匹配的、哪些区域尚未理解。该记录默认属于内部结构化状态，不要求默认落盘给最终用户。
17. 按需把 `findings.json`、`findings.md`、`texts.parquet`、`lines.parquet`、`line_groups.parquet`、`pair_candidates.parquet`、`pairs.parquet` 等信息持久化为调试/回归材料；正式产品运行时也可以只保留前端展示和必要导出所需的数据。

阶段 A 不直接输出最终错误结论。它只输出“证据”和“候选”。

阶段 A 产生的 findings / page findings / 转换缓存默认都属于会话级运行态。若用户未显式要求保留，应用关闭时应允许自动清理。

### 5.1.1 页级并行审图要求

系统应支持“按页并行分析”的工作方式。无论是人工审阅还是 Agent 辅助，默认粒度都应下沉到单张图纸，而不是只做项目级总览。

推荐要求如下：

- 一页图至少对应一个独立的页级 findings 记录。
- 页级 findings 可以并行产生，再汇总到项目级 findings。
- 页级 findings 的目标不是只写最终结论，而是写清“本页图是什么、结构是什么、识别器为什么这样选、当前能识别什么、还不能识别什么”。
- 页级 findings 默认属于内部运行态记录，不要求作为正式交付文件长期保留。
- 若为了调试而落盘，调试完成后应允许随时清理，不应在目录中无限累计历史中间产物。
- 若后续采用多 Agent 或多任务并发，最自然的分工方式应是“一页一任务”，避免多个任务同时在同一页上重复判断。
- 当用户点名某一张具体 DWG 说明新图型规则时，开发前必须先定位该原始 DWG、对应转换 DXF 或现有 findings，自行视觉核验或解析核验结构，再把规则写入任务书或实现；不能只根据口头描述直接改规则。

### 5.1.2 问题驱动的并发闭环要求

当真实样本仍存在明显 `issue_count`、`review` 洪峰或用户可见问题列表偏大时，主线程不得把“单次修一个点”视为结束，而应进入问题驱动的并发闭环。

推荐流程如下：

1. 主线程先重读任务书 active backlog、最新 audit 产物、最新真实样本统计和最近一次回归结果，确认当前剩余问题按“图类 / pair_kind / rule_id / root_cause”如何分布。
2. 主线程优先派发多个只读审计子代理，按页、按图类或按问题簇并发核验；审计子代理必须优先读原始 DWG、对应 DXF、findings、pairs、issues 和相关 parquet，而不是只汇报 `issue_count`。
3. 每个审计子代理都应输出最小闭环证据：`sheet_id`、`filename`、`current_output`、`expected_semantics`、`evidence`、`suspected_module`、`root_cause`、`suggested_fix_slice`。
4. 主线程汇总这些审计结论后，再派发开发子代理做最小修复切片；开发子代理的写集应尽量互不重叠，避免多个子代理同时改同一语义面。
5. 每轮修复后必须重新运行最贴近该切片的单测、集成测试和真实样本审计，对比 `issue_count`、`pair_kind` 分布、structured mapping 数量和残留伪普通 pair 是否下降。
6. 若问题下降但未清零，主线程应继续下一轮“审计子代理 -> 汇总 -> 开发子代理 -> 验证”的闭环，而不是停在单个 commit 或单个 phase 的完成提示上。

### 5.2 阶段 B：Audit & Report

输入：findings。
输出：`audit_report/` 目录。

流程如下：

1. 读取 `pairs.parquet` 和 `pair_candidates.parquet`。
2. 过滤低质量 pair，形成 high-confidence、review、discard 三类。
3. 建立 pair graph：数字为节点，配对为边，边上附带 sheet_id、line_id、confidence、evidence。
4. 运行规则引擎，发现冲突、缺失、重复、反向不一致和低置信度问题。
5. 为每个 Issue 生成解释：它为什么被认为有问题，来自哪几页，哪些候选被排除，冲突边是什么。
6. 输出 `issues.json`、`issues.xlsx`、`audit_report.md`、`audit_report.html`。
7. 前端展示异常清单，允许按严重程度、页码、数字、置信度、规则类型过滤。

## 6. Findings 数据规范

findings 是整个系统内部运行阶段的 Single Source of Truth。后续所有规则、报告和前端展示都必须基于 findings，但 findings 本身不是默认面向最终用户的正式交付物。

需要明确：

- 用户最终可见的核心结果是前端中的异常清单、证据、定位信息和必要导出。
- `findings` 更适合作为内部结构化状态、调试数据、回归材料和开发期分析依据。
- `findings` 可以按需持久化，但默认应视为可清理的运行缓存，而不是长期累计的项目资产。
- 应用退出或会话关闭时，若未开启调试保留模式，应自动清理 `findings`、`page_findings`、转换缓存和其他临时运行态数据。

若开启调试持久化，目录结构可参考：

```text
project_output/
  manifest.json
  findings/
    findings.md
    findings.json
    page_findings/
      S001.md
      S001.json
      S002.md
    pages.parquet
    texts.parquet
    blocks.parquet
    lines.parquet
    polylines.parquet
    line_groups.parquet
    terminal_candidates.parquet
    pair_candidates.parquet
    pairs.parquet
    extraction_warnings.parquet
  audit/
    issues.json
    issues.xlsx
    audit_report.md
    audit_report.html
  logs/
    ingest.log
    extract.log
    audit.log
  cache/
    converted_dxf/
    previews/
```

上述 `findings/` 与 `page_findings/` 均属于内部调试/回归材料，不是正式交付物。正式产品运行时，可以不默认保留这些文件，或在会话结束、调试完成后清理。

```json
  "page_no_source": "filename|title_block|manual|unknown",
  "confidence": 0.0,
  "warnings": []

```

除基础 Sheet 记录外，每张图还应有一份独立的页级 findings 记录，用于沉淀页级理解结果。它可以是内存对象、SQLite 记录，或按需落盘为 `page_findings/<sheet_id>.md` / `json`。该记录至少应包含：

- `page_type`
- `page_type_confidence`
- `layout_summary`
- `structure_summary`
- `recognition_strategy`
- `number_matching_strategy`
- `high_confidence_signals`
- `open_questions`

如果一页被判定为表格型图，则页级 findings 还应记录：

- 表格区域 bbox
- 行数、列数、列宽模式
- 是否命中“表头型三列表格”模式
- 识别到的表头前缀，例如 `1-21QD`
- 中间列行号序列是否从 `1` 开始连续递增
- 左/中/右三列分别承载什么文本角色：左侧接线端、行号、右侧接线端
- 每行合成出的逻辑语义端，例如 `1-21QD1`
- 每行逻辑语义端到左右两侧 `n###` 接线端的映射方式
- 为什么该映射可作为高置信信源

如果一页普通回路图被判定为元件分区型信号回路，则页级 findings 还应记录：

- 是否命中 `component_prefixed_signal_circuit` 子模式
- 每个元件分区的 bbox、分区 ID 和结构置信度
- 每个分区顶部识别到的前缀，例如 `1-2n`、`3-2n`
- 分区内局部数字及所属连接线，例如 `218`
- 外侧端子文本，例如 `1-4YD1`
- 合成逻辑端，例如 `1-2n218`
- 每条映射的完整证据链：分区前缀文本、局部数字文本、外侧端子文本、连接线 ID、坐标和分区 bbox
- 哪些连接因前缀缺失、局部数字不唯一、外侧端子不唯一或线段归属不清而只能进入 review

如果一页普通回路图被判定为线中元件端口型回路，则页级 findings 还应记录：

- 是否命中 `inline_component_port_circuit` 子模式
- 每个线中元件的 bbox、本体编号和结构置信度，例如 `3-2KLP1`
- 元件左右端口序号及其坐标，例如 `1`、`2`
- 左侧外部端子文本，例如 `3-2QD2`
- 右侧局部数字及其所属页/区域前缀，例如 `116` + `3-2n`
- 合成逻辑端，例如 `3-2n116`
- 每条映射的完整证据链：元件本体编号、端口序号、外侧端子或合成端子、连接线 ID、坐标和元件 bbox
- 哪些连接因元件本体编号缺失、端口序号不唯一、右侧前缀缺失或线段归属不清而只能进入 review

如果一页普通回路图被判定为测控开入矩阵型回路，则页级 findings 还应记录：

- 是否命中 `input_matrix_wire_mapping` 子模式
- 每个矩阵列的 bbox、列 ID、顶部前缀和结构置信度，例如 `1-21n`
- 每个水平行带的 row_id、y 坐标范围和所属列
- 同一行内的完整端子文本，例如 `1-21QD12`
- 同一行内的裸数字端，例如 `127`
- 裸数字端补全后的逻辑端，例如 `1-21n127`
- 每条映射的完整证据链：列前缀文本、完整端子文本、裸数字文本、行带 ID、列 ID、坐标和同一行/同一列判定依据
- 被旁路的中文/英文说明文本，例如 `开入32/对象6分位`、`BI 32/Object 6 Opening position`
- 哪些行因列前缀缺失、完整端子缺失、裸数字多义、行带归属不清或说明文本抢占候选而只能进入 review

如果一页被判定为背板表格型图，则页级 findings 还应记录：

- 原始 `sheet_category` 是否为背板图，以及为什么仍升级为表格型审计
- 是否需要展开块内虚拟文本，以及命中的 `INSERT` 块名，例如 `WBH-814E-E1SA-101`
- 原始表头文本，例如 `NKR308A(非电量选配)`
- 规范化后的表头前缀，例如 `NKR308A`
- 被旁路的中文说明文本，例如 `非电量选配`、`调压重瓦斯开入`
- 行号文本及其规范化端口序号，例如 `01 -> 1`
- 外侧接线端文本，例如 `5FD15`
- 每行输出的背板表格映射，例如 `NKR308A-1 -> 5FD15`
- 哪些行因行号缺失、外侧接线端缺失、左右列不对齐或表头不唯一而只能进入 review

如果一页被判定为元件接线图，则页级 findings 还应记录：

- 是否命中“分离独立子元件实例”模式
- 若是 `23 元件接线图3.dwg` 这类长条双端口结构，是否命中 `strip_two_port_component` 子模式
- 若是 `AK/A'/KZKK/JR` 这类小型端口盒结构，是否命中 `small_port_box_component` 子模式
- 每个子元件实例的 bbox、实例 ID 和结构置信度
- 每个子元件最上方圆形编号区识别到的本体编号，例如 `1-40DK2`
- 每个子元件识别到的输出数量或结构类型，例如 `4输出`、`6输出`、长条双端口、小型端口盒
- 端口序号区域和端口序号列表，例如 `1..4` 或 `1..6`
- 每个端口合成出的逻辑端，例如 `1-40DK2-1`
- 每个端口对应的外部接线端，例如 `ZD17`
- 对长条双端口结构，还应记录块实例名、两端圆形端口、端口 `1/2` 的坐标，以及每端对应的外部端，例如 `5KLP10-1 -> 5KLP9-1`、`5KLP10-2 -> 5n112`
- 对小型端口盒结构，还应记录圆形本体编号、端口盒 bbox、端口格布局、端口号坐标、外侧接线端坐标，以及每个端口映射，例如 `KZKK-1 -> JD8`、`KZKK-2 -> K-5`、`JR-1 -> K-3`、`JR-2 -> K-4`
- 每条组件映射的证据链：本体编号文本、端口序号文本、外部接线端文本、坐标和所属子元件实例
- 哪些子元件因本体编号缺失、输出数量缺失、端口数量不连续或外部接线端不唯一而只能进入 review

若这些页级 findings 被临时写入磁盘，它们应被视为可清理的中间结果，而不是正式交付目录中的长期资产。

## 7. 几何识别策略

### 7.1 有效审计区识别

第一版优先使用规则和配置结合。

系统应先从所有线段中寻找最大外框，推断图框 bbox。若图纸存在标题栏，通常位于下方或右下方，系统可以通过密集短线、表格线、文字密集区识别。若自动推断失败，则用户可在配置文件中写入 `audit_area_bbox`。

封面、目录、屏面布置图、标牌图等非配线页应可被跳过。跳过方式有三种：文件名规则、页标题规则、人工配置。

### 7.2 水平线识别

候选连接线需满足：

角度接近 0° 或 180°，默认容差 ±2°。
长度超过最小阈值，默认 20 图纸单位。
位于有效审计区内。
不是图框线、标题栏线、表格线。
左右端附近存在数字文本候选。

线段合并策略：

若两条线 y 坐标差小于 `line_y_tolerance`，x 方向间隙小于 `line_gap_tolerance`，角度差小于 `angle_tolerance`，且图层相同或图层相容，则合并为 LineGroup。

### 7.3 数字候选识别

候选数字默认只接受以下模式：

纯数字：`101`、`12`、`203`。
可配置允许前缀/后缀：如 `X1:101`、`101A`、`1-01`。MVP 默认关闭复杂模式。
过滤页码、标题栏编号、图名编号、设备型号编号。

数字候选评分由以下因素组成：

文本是否纯数字。
是否位于有效审计区。
是否在连接线端点搜索窗口内。
与端点的垂直偏差是否小。
是否在线端正确方向，例如左端数字应位于左端附近，右端数字应位于右端附近。
文字高度是否落在端子数字常见范围。
图层是否属于文字/端子/标号图层，而不是标题栏图层。
附近是否存在竞争数字。

### 7.4 普通 Pair 黑名单与语义旁路

不是所有数字或短文本都应该进入普通 pair 候选池。系统必须显式区分“真实端子/映射数字”和“语义标签/说明性文本/延续标记”。

至少应支持以下策略：

- 对明显语义标签、说明词、相别代号、功能代号、单位电压标注等内容，默认不进入普通 pair。
- 对端子图中的 `DK/KLP/ZKK/CLP`、相别类 `UA/UB/UC/UN/3U0`、说明性列头、屏蔽层文字等，应优先走语义旁路或专用通道，而不是与真实端子号竞争同一个 rank。
- 对单字符数字、块内固定引脚号、尺寸号、局部序号等高噪声来源，必须允许按页型、图层、块名和区域进行强降权或剔除。
- 对元件分区型普通回路图中的局部数字，例如 `218`、`221`，默认不能裸用为跨页端子号；只有与所属分区前缀组合成 `1-2n218` 这类完整逻辑端后，才允许进入审计。
- 对线中元件端口型普通回路图中的端口序号 `1`、`2`，默认不能裸用为跨页端子号；只有与同一线中元件本体编号组合成 `3-2KLP1-1`、`3-2KLP1-2` 这类完整元件端口后，才允许进入审计。右侧局部数字如 `116` 也必须结合页/区域前缀合成为 `3-2n116` 或 `1n116` 后才能参与审计。
- 对测控开入矩阵型普通回路图中的裸数字，例如 `127`、`212`、`228`，默认不能裸用为跨页端子号；只有与所属矩阵列前缀组合成 `1-21n127`、`1-21n212`、`1-21n228` 这类完整逻辑端后，才允许进入审计。
- 对元件接线图中的端口序号 `1..4`、`1..6`，以及长条双端口结构中的圆内端口 `1`、`2`，默认不能作为普通端子号独立进入 PairBuilder；它们只有在同一子元件实例内与本体编号组合成 `1-40DK2-1`、`5KLP10-1` 这类完整逻辑端后，才允许作为组件映射参与审计。
- 对背板表格型图中的中文功能说明和括号说明，默认进入 `semantic_channel` 或说明证据，不能作为匹配目标覆盖表头、行号和外侧接线端。
- 被旁路的内容不是“丢弃”，而是保留为可解释证据，供页级 findings、语义映射和人工复核使用。

建议把候选先分到四个内部通道：

- `terminal_numeric_channel`：允许进入普通 pair 的真实端子数字候选。
- `semantic_channel`：语义标签、相别、功能代号、说明列等。
- `continuation_channel`：延续、续接、单侧去向、桥接提示等。
- `noise_channel`：尺寸号、局部序号、块内固定引脚号和其他高噪声对象。

默认约束如下：

- PairBuilder 只直接消费 `terminal_numeric_channel`。
- `semantic_channel` 与 `continuation_channel` 可以生成独立记录，但不能与普通端子候选抢同一个 winner。
- `noise_channel` 默认不参与配对，只保留可解释证据。

### 7.5 页型主信源定义

不同页型的主信源必须明确，不能继续用一套“近邻文本”策略覆盖所有页面：

- 普通回路图：主信源是导线路径两端与线身上的数字关系，核心约束是“数字属于这条线”，而不是“数字离这条线很近”；若命中元件分区型信号回路，则还必须使用分区顶部前缀合成完整逻辑端；若命中线中元件端口型回路，则必须使用线中元件本体编号和端口序号合成完整元件端口。
- 元件接线图：主信源是分离子元件实例、圆形本体编号、输出数量或长条双端口结构、端口序号和外部接线端之间的结构化组合关系；必要时再辅以块内虚拟文字、标签、导线和方向感知。
- 端子图：主信源是行锁定、列角色、代号和同行对应关系。
- 表格型图：主信源是表格结构与单元格映射，不要求先还原导线。
- 背板表格型图：主信源是块内插件/端子表格结构、规范化表头、行号和外侧接线端；中文说明只作为解释证据。

## 8. 置信度模型

置信度必须是可解释的，不允许只给一个黑盒分数。

Pair confidence 建议按以下方式计算：

```text
pair_confidence =
  0.25 * left_candidate_score +
  0.25 * right_candidate_score +
  0.20 * line_geometry_score +
  0.15 * uniqueness_score +
  0.10 * layer_context_score +
  0.05 * project_pattern_score
```

字段含义：

`left_candidate_score`：左端数字候选质量。
`right_candidate_score`：右端数字候选质量。
`line_geometry_score`：线段是否水平、是否足够长、是否由稳定线段合并。
`uniqueness_score`：每端是否只有一个明显候选。
`layer_context_score`：线和文字图层是否符合项目规律。
`project_pattern_score`：该 pair 是否符合整套图中反复出现的稳定模式。

置信度桶：

`high`：>= 0.92，可进入自动一致性判断。
`review`：0.75–0.92，不作为 hard error，但进入人工复核队列。
`low`：< 0.75，不参与最终规则判断，只保留证据。

Issue confidence 不等于 Pair confidence。Issue confidence 应考虑冲突 pair 的置信度、冲突数量、是否来自不同页、是否存在替代候选。

```text
issue_confidence =
  min(confidence(pair_a), confidence(pair_b))
  * conflict_strength
  * evidence_completeness
```

## 9. 规则引擎设计

规则引擎必须插件化，每条规则独立输入 findings 或 pair graph，输出 Issue。规则必须有 `rule_id`、`name`、`description`、`severity_default`、`input_tables`、`parameters`、`output_issue_type`。

进入规则引擎前，系统必须先把页面内关系分成至少七类：

- `ordinary_pair`：普通端子到端子配对。
- `wire_component_mapping`：普通回路图中的结构化导线映射，包括“元件分区前缀 + 局部数字 -> 外侧端子”、“线中元件本体编号-端口序号 -> 外侧端子/合成端子”和“测控开入矩阵完整端子 -> 列前缀补全裸数字端”。
- `component_mapping`：元件接线图中“本体编号-端口序号 -> 外部接线端”的组件端口映射，覆盖 `4输出/6输出` 子元件和长条双端口子元件。
- `table_mapping`：表格型图中“表头前缀 + 行号 -> 接线端”的单元格映射。
- `continuation`：延续、续接、单侧待去向记录。
- `bridge_mapping`：桥接带、短接列或局部跨列映射。
- `semantic_mapping`：端子到语义端、代号列、标签列的映射。

其中 `continuation` 与 `bridge_mapping` 不能直接等价于普通 pair。例如 `? -> 328`、`110 -> 330`、`10 -> ?` 更可能是续接或桥接记录，需要专用解释；如果系统把它们直接塞进普通 pair，将系统性制造伪冲突与伪缺失。

MVP 默认语义建议如下：

- `ordinary_pair`：只有当两侧都命中真实端子数字通道，且关系属于同一导线路径或同一行锁定时，才允许进入普通跨页一致性校验。
- `wire_component_mapping`：默认代表普通回路图中由结构上下文解析出来的稳定导线关系，例如 `1-2n218 -> 1-4YD1`、`3-2KLP1-1 -> 3-2QD2`、`3-2KLP1-2 -> 3-2n116`、`1-21QD12 -> 1-21n127`。它可作为高置信结构化信源参与跨页一致性校验，但报告中必须保留分区前缀、线中元件本体编号或矩阵列前缀、局部数字或端口序号、外侧端子/完整端子、连接线或行带证据。
- `component_mapping`：默认代表元件接线图内稳定子元件端口关系，例如 `1-40DK2-1 -> ZD17`、`5KLP10-1 -> 5KLP9-1`、`5KLP10-2 -> 5n112`。它应作为高置信结构化信源参与跨页一致性校验，但报告中必须保留子元件实例、本体编号、端口序号、外部接线端和块/几何结构证据。
- `table_mapping`：默认代表表格型图内稳定单元格映射，例如 `1-21QD1 -> 1-21n552`。它应作为高置信结构化信源参与跨页一致性校验，但报告中必须保留表头、行号、单元格和源文本证据。
- `continuation`：默认代表“单侧已知、另一侧待续接或待去向”的记录。它应进入 review 证据，但不应单独制造 hard conflict。
- `bridge_mapping`：默认代表端子页桥接带、短接列或局部列间映射。它可以参与页内解释和后续统一性校验，但不应直接等价为普通端子对端子配对。
- `semantic_mapping`：默认代表端子到语义端、代号列或标签列的对应关系。它应单独建索引，用于“端子号对语义端一致性”校验。

只有在以下情况下，`continuation` 或 `bridge_mapping` 才允许升级成更强问题：

- 与同页或跨页稳定高置信普通关系矛盾。
- 同一编号在多个 continuation / bridge 记录中互相冲突。
- 按项目规则本应闭合，却长期找不到对应去向。

### R-PAIR-MISSING-SIDE

含义：一条候选连接线只有一端找到数字，另一端没有可靠数字。

触发条件：LineGroup 存在，但 left 或 right 端没有 rank=1 且 score >= 阈值的 TerminalCandidate。

输出：review 或 major，取决于线段置信度。

### R-PAIR-LOW-CONFIDENCE

含义：左右端都有数字，但 pair confidence 在 review 区间。

触发条件：0.75 <= pair_confidence < 0.92。

输出：review。

### R-DUPLICATE-SAME-LINE

含义：同一条线一端出现多个竞争数字，且分数接近。

触发条件：同一 side 的 top1 与 top2 分差小于 0.08。

输出：review。

### R-CROSS-PAGE-CONFLICT

含义：同一数字在不同页面上对应了不同目标数字。

例子：S08 中 101--202，S14 中 101--203。

触发条件：以无向 pair graph 分析，一个数字节点连接多个不同目标，且相关边均 high confidence。

输出：critical。

### R-ONE-TO-MANY

含义：一个数字被多个数字配对，且这些配对不属于可配置允许的一对多模式。

触发条件：节点 degree > 1，且所有边不在白名单规则内。

输出：major 或 critical。

### R-MANY-TO-ONE

含义：多个数字指向同一数字，可能代表汇接，也可能代表错误。MVP 默认进入 review，不直接判错。

触发条件：若采用有向 pair，多个 source 指向同一 target。

输出：review。

### R-MISSING-RECIPROCAL

含义：某个跨页配对在 A 页出现，但在 B 页找不到反向或相应配对证据。

触发条件：配置要求 reciprocal=true 时启用。

输出：major。

### R-DUPLICATE-PAIR

含义：同一 pair 在多处重复出现。是否异常取决于项目规则。

触发条件：相同 unordered pair 出现次数超过阈值，且不属于允许重复类型。

输出：review。

### R-SHEET-PAGE-MISMATCH

含义：文件名页码、标题栏页码和图内页码不一致。

触发条件：`filename_page_no != title_block_page_no`。

输出：major。

## 10. 前端设计

当前 Streamlit UI 仅用于内部验证与开发调试，不作为最终交付形态。

最终交付形态应为本地 Windows `exe` 校验客户端。前端不直接实现几何推理，只负责任务编排、过程可视化、结果复核和最近项目回看。

### 10.1 产品定位

1. 本工具的产品定义是本地一次性校验工具，不要求用户管理输出目录。
2. 应用内部仍可使用临时工作目录保存转换、抽取和预览过程产物，但这些目录对用户默认隐藏，任务完成后允许自动清理。
3. 用户默认可见交付应收敛为异常清单、定位证据和审计报告，而不是要求用户理解内部 findings 目录。
3. 用户侧长期保留的数据以最近项目和最终问题结果为主，不要求长期保留全部 DXF、预览图和中间 findings 文件。
4. UI 只能调度本地任务并展示结果，不得在前端重复实现 CAD 几何推理。

### 10.2 页面结构

页面一：启动页。
功能：显示最近项目、拖拽导入项目目录、点击选择目录、检测输入有效性、点亮“开始分析”按钮。

页面二：过程页。
功能：显示当前阶段、当前页数 / 总页数、进度条、实时日志、实时发现的问题表格，并承担一部分结果预览职责。

页面三：结果页。
功能：显示最终问题表格、问题详情、置信度拆解、定位预览图和红框标注。

设置页不单独作为一级导航。配置项优先使用弹窗或侧边抽屉。

### 10.3 启动页要求

1. 显示最近项目列表，最近项目点击后直接进入该项目的最终结果页。
2. 支持拖拽项目目录导入，也支持按钮选择目录。
3. 用户未选择有效输入前，“开始分析”按钮保持禁用。
4. 不向用户暴露输出目录选择；桌面端默认使用应用内部临时工作目录。
5. 启动页可显示最近一次校验时间、图纸页数和问题数量摘要。

### 10.4 过程页要求

1. 页面顶部显示阶段状态：扫描、转换、抽取、配对、审计、渲染预览、完成。
2. 页面中部显示总进度条，并至少展示当前页数 / 总页数、已完成页数、失败页数和累计发现问题数。
3. 页面底部或右侧显示实时日志流，便于定位转换失败、抽取异常和规则错误。
4. 过程页需要实时刷新问题表格，而不是等待全量结束后再统一展示。
5. 实时问题表格字段至少包括：页码、文件名、规则、标题、左右数字、问题类型、置信度、当前状态。
6. 发现高风险问题时可使用轻量提示，但不应使用阻塞式弹窗打断流程。
7. 过程页本身承担一部分结果页职责，用户可以在任务未结束时先看到已发现的问题和对应页信息。

### 10.5 结果页要求

1. 左侧显示最终问题表格，支持按页码、规则、严重程度、状态和左右数字筛选。
2. 右侧显示当前问题的图样预览，并在对应区域使用红框或高亮框标出疑点位置。
3. 结果页必须显示坐标、线 ID、页码、文件名、关联数字、候选排名和证据链。
4. 若当前问题涉及多个候选或跨页冲突，应支持显示相关 pair 和关联页列表。
5. 若源 DWG 仍可访问，结果页允许按需重新生成当前页预览；若源文件不可访问，则至少保留问题表格与文字证据。

### 10.6 图样预览与渲染

1. 第一版结果页不实现完整 DWG 浏览器，但必须支持当前问题所在页的局部或整页预览。
2. 预览可以来自临时 PNG、SVG 或其他轻量渲染结果，重点是支持坐标定位与红框标注。
3. 预览产物默认视为临时文件，不要求长期保留；如用户再次打开最近项目且源文件仍存在，可按需重新生成。
4. 如果后续引入更高保真渲染引擎，应保持问题定位字段与 UI 展示契约不变。

### 10.7 知信度与业务判定

知信度不等于业务错误严重程度。系统必须将“机器识别把握”与“业务判定类型”分开保存和展示。

推荐采用三层表达：

1. `terminal_conf`：某个端点数字候选的识别把握。
2. `pair_conf`：某条候选连接线两端数字配对的整体把握。
3. `issue_conf`：某条审计问题结论的可信程度。

推荐的首版拆分方式：

1. `terminal_conf` 由垂直对齐、左右侧一致性、文本类型、文字高度等子项组成。
2. `pair_conf` 由线段候选分、左右端点候选分和候选分差共同组成。
3. 候选分差必须单独保留，因为“第一名与第二名非常接近”通常比单个原始分数更能说明歧义。
4. 结果页展示时，不能只给一个总分，必须能拆解显示影响该分数的主要因素。

推荐的首版权重草案：

```text
terminal_conf =
  0.35 * vertical_alignment_score +
  0.35 * horizontal_side_score +
  0.15 * text_type_score +
  0.15 * height_score

pair_conf =
  0.30 * wire_candidate_score +
  0.25 * left_terminal_conf +
  0.25 * right_terminal_conf +
  0.20 * candidate_gap_score
```

上述权重只是首版草案，必须通过样本回归和人工标注继续校准，不能视为最终固定参数。

### 10.8 一对多与分支语义

1. 同一数字在业务上可能合法对应两个端子或两个引用目标，因此“一对多”不能默认视为错误。
2. 系统必须将“一对多”拆分为三类：
   - `branch`：合法分支，不记为错误，但需要可见。
   - `review`：存在一对多，且候选分差小、反向关系弱或证据不足，需要人工复核。
   - `conflict`：同一数字跨页或跨上下文指向明显矛盾目标，才视为真正冲突。
3. 规则引擎和结果页都必须支持把“一对多”从纯报错升级为“结构现象 + 风险分级”的表达方式。

### 10.9 最近项目与本地存储

桌面端引入轻量 SQLite，用于保存最近项目和最终问题结果，而不是长期保留整套中间工件。

SQLite 至少保存以下信息：

1. 项目路径、项目名称、最近校验时间、图纸总页数、问题总数。
2. 每条 issue 的页码、文件名、规则编号、标题、左右数字、严重程度、状态、知信度。
3. 用于结果定位的最小证据字段，例如线 ID、坐标、bbox、关联 pair、evidence JSON。
4. 最近项目点击后必须能直接打开最终问题表格；如果源图纸还在，则允许按需重建预览图。

### 10.10 前端技术路线

1. 内部 MVP：Streamlit，用于流程验证和研发调试。
2. 正式产品：Tauri 2 + React + TypeScript，本地打包为 Windows `exe` 客户端。
3. 桌面端优先使用 Tauri，而不是 Electron，以降低资源占用和安装体积。
4. 正式产品阶段不要求先实现独立 HTTP 后端。

### 10.11 前端任务拆分

1. 搭建 Tauri 2 + React + TypeScript 桌面壳。
2. 实现启动页：最近项目、拖拽导入、目录选择、输入校验、开始分析。
3. 实现过程页：阶段状态、页数进度、实时日志、实时问题表格。
4. 实现结果页：问题表格、详情抽屉、置信度拆解、红框定位预览。
5. 实现桌面端与 Python sidecar 的任务编排与事件流通讯。
6. 实现轻量 SQLite 数据存储，仅保存最近项目和最终问题结果。

### 10.12 企业桌面交互与视觉标准

1. 正式桌面端应呈现为企业工作台，而不是演示型玩具界面。
2. 视觉风格应偏克制、稳定、高信息密度，强调表格、状态、筛选、详情与定位，而不是大块说明文案或装饰性卡片。
3. 启动页重点是项目导入与最近项目回看，不做营销式首页。
4. 过程页重点是阶段状态、页级进度、失败项、警告、实时问题表格与日志，不做无业务价值的装饰。
5. 结果页重点是“问题表格 + 详情证据 + 预览定位”的高效布局，便于连续复核。
6. 所有用户常用操作应在桌面端直接完成，不要求用户理解输出目录、临时工件或 CLI 参数。
7. 前端文案应贴近企业审计工具口径，减少样板式提示语和开发者口吻。

## 11. 后端 API / CLI

必须先实现 CLI，再实现前端。CLI 是系统自动化和 Agent 调用的基础。

### CLI 命令

```bash
dwg-audit init-config
dwg-audit analyze-project --input ./project_dwg --output ./out --config ./config.yml
dwg-audit export-findings --output ./out
dwg-audit run-audit --findings ./out/findings --output ./out/audit
dwg-audit export-report --audit ./out/audit --format xlsx,html,md
dwg-audit serve --project ./out
```

桌面客户端阶段不强制要求独立 HTTP API。正式产品优先采用“Tauri command + Python sidecar + 本地文件 / SQLite”的方式集成。

### 11.1 桌面集成原则

1. 现有 Python CLI 和 pipeline 是本地业务引擎，应尽量复用，不应为了桌面端另起一套独立后端服务。
2. Tauri 层负责目录选择、任务启动、事件转发、结果查询、最近项目管理和文件打开。
3. Python sidecar 负责 DWG 扫描、转换、抽取、配对、审计、预览渲染和结果落库。
4. 前端不能直接操作 DXF 几何对象；它只能消费 sidecar 输出的事件、SQLite 结果和最小定位数据。

### 11.2 Python sidecar 契约

桌面端应在现有 CLI 之外补充适合交互式任务的 sidecar 契约。推荐支持：

```text
analyze-session --input <project_path> --session-id <id>
load-result --project-id <id>
render-preview --project-id <id> --sheet-id <id> --issue-id <id>
list-recent-projects
purge-session --session-id <id>
```

运行中的 sidecar 需要持续输出结构化事件，至少包括：

```text
progress
page_started
page_finished
issue_found
warning
run_finished
run_failed
```

事件建议采用 JSONL 或等价的结构化文本格式，便于 Tauri 实时转发到过程页。

### 11.3 可选 HTTP API（后续阶段）

```text
POST /api/projects/analyze
GET  /api/projects/{project_id}/summary
GET  /api/projects/{project_id}/findings/pages
GET  /api/projects/{project_id}/findings/pairs
GET  /api/projects/{project_id}/issues
GET  /api/projects/{project_id}/issues/{issue_id}
POST /api/projects/{project_id}/issues/{issue_id}/status
GET  /api/projects/{project_id}/export/issues.xlsx
GET  /api/projects/{project_id}/export/report.html
```

桌面版 MVP 优先使用 SQLite + 本地文件，不引入数据库服务。后续若要支持多项目、多用户、多版本与远程协作，再升级为 PostgreSQL 或其他服务化存储。

### 11.4 当前阶段 CLI 与产品边界

1. CLI 仍保留为内部业务引擎入口、自动化入口、测试入口和 sidecar 调用基础，不作为最终用户主界面。
2. 当前阶段不再以“新增 CLI 命令数量”衡量进展；新增终端命令只有在服务测试、回归、打包、调试或 sidecar 契约时才有意义。
3. 正式产品能力应优先收口到“Tauri 桌面工作流 + Python sidecar + 本地状态存储”，而不是继续把用户功能堆到 CLI。
4. `evaluate-acceptance-suite`、回归比较、故障注入等命令保留为内部 harness，不作为产品功能暴露。

## 15. 测试方案

测试必须分四层。

第一层是单元测试。测试文本归一化、bbox 计算、角度计算、线段合并、候选评分、规则触发。

第二层是集成测试。用 2–3 张小型 DWG/DXF 验证从 ingest 到 report 的完整链路。

第三层是 golden regression。固定一套样本图，保存 expected findings 和 expected issues。每次修改后比较关键字段差异。

第四层是人工抽样评估。随机抽取若干页，人工标注线端数字 pair，计算系统的召回率、准确率和低置信度覆盖率。

第五层是隔离故障注入测试。由于两套真实样本项目默认视为正确图纸，因此冲突、缺失和错配的硬错误验证，不应只依赖真实样本“跑出错误”，而应通过隔离副本、变异样本或合成夹具人为改动端子号来验证。

建议指标：

文本数字召回率 >= 98%。
目标水平线召回率 >= 95%。
高置信 Pair 准确率 >= 97%。
跨页冲突 hard error 误报率尽可能低，宁可进入 review。
所有低置信、模糊、多候选场景必须进入 review，不得静默忽略。
两套真实样本项目在默认配置下，不应被批量判定为大量 hard error；若出现大量 critical/major，必须先回到 findings 和页型策略定位原因。
隔离故障注入样本中，人工制造的冲突、缺失、错映射应被稳定命中，并与真实样本评估分开汇报。

## 16. 风险与应对

风险一：DWG 转 DXF 后对象保真度不足。
应对：Reader Adapter 抽象必须提前设计。MVP 使用 odafc，后续可切换 ODA Drawings SDK 或 RealDWG。

风险二：文字不是 TEXT/MTEXT/ATTRIB，而是被炸成线段。
应对：MVP 标记为 extraction warning；后续引入 OCR/VLM fallback。

风险三：图纸中有大量表格线、图框线被误识别为连接线。
应对：标题栏与图框过滤、线长分布分析、端点数字存在性过滤、图层过滤。

风险四：同一个数字天然允许一对多。
应对：规则引擎必须支持白名单和项目配置。默认保守进入 review，不直接判错。

风险五：坐标定位对用户不直观。
应对：第一版给页码、文件、坐标；第二版增加局部截图或 SVG 片段；第三版再考虑 DWG viewer。

风险六：误报导致用户不信任。
应对：所有 issue 必须给证据链、候选列表、排除理由和置信度。不要只输出“错误”。

风险七：Agent 每次重新读图导致结果不可复现。
应对：findings 作为 Single Source of Truth。规则迭代只能基于 findings 重跑，除非明确执行重新读图。

## 18. 给本地 Agent 的执行指令

本地 Agent 开发时必须遵守以下顺序：

第一步，不要先做 UI。先实现 CLI 和 findings。
第二步，不要先写复杂规则。先证明 DWG 可以稳定读成结构化对象。
第三步，不要只保存最终 Pair。必须保存所有候选和评分。
第四步，所有规则必须基于 findings，不允许规则模块直接读取 DWG。
第五步，每个异常必须能解释“为什么报错”。
第六步，所有低置信度情况进入 review，不允许强判。
第七步，每次优化算法后，必须用同一套 findings 重跑规则，比较结果变化。
第八步，只有当 findings 本身错误时，才重新进入读图阶段。
第九步，若采用多 Agent 或角色分工，建议把“变异样本制作”“执行测试”“结果复核”分开，避免同一实现链路既制造测试又单方宣告通过。

### 18.1 当前阶段补充指令（产品化收口阶段）

1. 当 M0-M8 与主要抽取 / 规则主链已基本成立后，不得再以扩 CLI 代替桌面产品闭环。
2. 主线程优先负责任务编排、状态刷新与阶段收口；只读审计、前端视觉重构、运行时验证可尽量交给独立测试会话或独立子代理。
3. 产品层工作与规则层工作允许并行，但不得混淆“业务引擎问题”和“产品编排问题”。
4. 已闭环的抽取器能力不应被反复重开；当前剩余抽取红线、产品前端体验、sidecar / runtime、exe 交付应分轨推进。
5. 正式产品默认只暴露桌面工作流，不暴露 CLI 思维模式给最终用户。

### 18.2 当前阶段主线程夜间自迭代要求

1. 主线程在完成当前最小切片后，不得因“本轮任务已完成”即停止；只要两套真实正确样本的默认用户视角问题列表仍明显大于 0，就应继续下一轮闭环。
2. 每一轮闭环都应先派发多个只读审计子代理，对剩余问题按页、按图类、按问题簇并发读图和读产物归因；禁止只根据 issue 汇总数字直接改规则。
3. 审计子代理必须优先回答“问题所在图纸上原始结构是什么、当前输出为什么错、哪一个模块最可能导致该问题、最小修复切片是什么”，并提供可回查证据。
4. 主线程收到审计结论后，再并发派发开发子代理做最小修复；开发子代理默认只改一个清晰 root cause，不把多个语义面揉进同一切片。
5. 每轮修复后，主线程必须组织验证：针对性单测、集成测试、真实样本审计、前后 issue 对比，以及是否引入新的回退。
6. 若某类问题在真实正确样本上本应消失，则目标是继续压降直到默认用户视角不再看到它；若某类输出只是结构现象或解释证据，则应改造分类和展示方式，而不是长期保留为默认问题项。
7. 主线程应持续把新的确定性问题、已闭环问题、剩余问题簇和下一轮最小切片写回任务书或配套进度文件，保证长时间自治过程中不会跑偏。

## 19. 最小验收场景

验收必须同时包含“真实正确样本”和“隔离故障注入样本”两部分。

真实正确样本可直接使用当前两套项目，它们默认被视为正确图纸。隔离故障注入样本应从副本或合成夹具生成，不能直接污染真实项目。

准备一套 5 张 DWG 的小样本，其中包含：

正常跨页匹配 10 组。
至少 1 张元件分区型普通回路图，包含 `1-2n218 -> 1-4YD1` 这类分区前缀映射样例。
至少 1 张线中元件端口型普通回路图，包含 `3-2KLP1-1 -> 3-2QD2`、`3-2KLP1-2 -> 3-2n116` 这类线中元件端口映射样例。
至少 1 张元件接线图，包含多个子元件实例，并有 `4输出` / `6输出` 端口映射样例。
至少 1 张长条双端口元件接线图，包含 `5KLP10-1 -> 5KLP9-1`、`5KLP10-2 -> 5n112` 这类块展开端口映射样例。
至少 1 张表头型表格图，包含 `表头前缀 + 行号 -> n### 接线端` 映射样例。
至少 1 张背板表格型图，包含块内虚拟表头、中文说明旁路、行号规范化和外侧接线端映射样例。
故意制造数字冲突 2 组。
故意制造缺失匹配 2 组。
故意制造一端多个候选数字 2 组。
包含一张非回路页。

验收时要求：

系统能跳过非回路页。
系统能抽出至少 90% 的人工标注 Pair。
故意制造的 2 组冲突必须进入 issue。
多个候选数字必须进入 review，而不是直接判错。
报告中每个 issue 都能定位到文件名、页码、坐标和相关线。
真实正确样本不应被系统稳定打出明显错误的 hard error 洪峰；如出现，应优先视为算法问题而不是图纸问题。

## 20. 最终 MVP 定义

MVP 不是一个完整 CAD 软件，也不是一个电气图纸智能理解系统。MVP 是一个本地离线审计工具，核心能力是：

读取一套 DWG；
内部生成可追溯 findings 运行态；
按页分类并路由到专用识别器；
抽取普通回路图、元件接线图、端子图、表格图中的有效映射；
建立端子对端子、普通回路元件分区对外侧端子、端子对语义端、元件端口对接线端、表格对接线端五类关系；
跨页校验上述关系的一致性；
输出可复核异常清单、定位证据和审计报告；
用置信度和证据链降低误判风险。

这一路线最贴近当前需求，也最贴近成熟电气 CAD 产品的工程思想：项目级对象、跨页引用、规则检查、消息清单、可跳转定位和人工复核优先，而不是黑盒识图优先。
