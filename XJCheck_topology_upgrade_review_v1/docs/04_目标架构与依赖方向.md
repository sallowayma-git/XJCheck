# 目标架构与依赖方向

## 1. 分层依赖

```text
UI/Report
   ↓
Audit V2 / Witness
   ↓
Project Cross-page Graph
   ↓
Constraint Solution
   ↓
Semantic Attachments + Electrical Nets
   ↓
Symbol-Port Graph + Geometry Graph
   ↓
CAD Primitive Model
   ↓
Reader Adapter + Project/Sidecars
```

上层不得反向修改下层事实。规则不能重新读取 DWG，UI 不能重新推理。

## 2. 模块边界

### Reader

负责把不同后端统一成 `CadDocument`。不做电气判断。

### Primitive

负责 WCS、block nesting、entity provenance。不得使用端子正则。

### Topology

负责几何关系和四态决策。不得包含 KLP/ZKK 等业务 family。

### Symbols

负责块族、端口、内部连接、文字槽。

### Networks

只对 ASSERTED connectivity 物化网络。

### Semantics

负责文字角色、端子 identity 和 scope。

### Inference

负责候选排序、约束和可选模型。

### Audit

负责项目级规则和 Issue。

## 3. 推荐依赖

核心：

- ODA File Converter / ODA Drawings SDK 或 RealDWG；
- ezdxf；
- Shapely STRtree；
- NetworkX（MVP/解释）或 rustworkx（后续大图性能）；
- OR-Tools CP-SAT；
- pandas/pyarrow；
- pydantic/jsonschema；
- SVG 生成（ezdxf drawing add-on 或自定义）；
- pytest/hypothesis。

可选学习：

- scikit-learn；
- LightGBM/XGBoost；
- PyTorch；
- PyTorch Geometric。

## 4. 数据存储

MVP 继续使用 Parquet + JSON，增加 SQLite 作为索引和 UI 状态库即可。不要在几何主链尚未稳定时引入复杂分布式数据库。

建议：

```text
Parquet: 大型 findings 表
JSON: manifest/schema/decision trace
SQLite: project index, issue state, review labels, run metadata
YAML: project profile, symbol library, rule scheme
```

## 5. 性能原则

- 空间查询必须走 STRtree/R-tree；
- 相交计算按 page/zone/bbox 分块；
- block definition 指纹按 definition 缓存；
- 只在交点处切分，不对所有线全量两两比较；
- Reader cache 由 file hash + backend version 控制；
- Rule rerun 不重新读取 DWG；
- 模型特征由 findings 构建，可离线复用。
