## 4. `.prj` 可恢复的弱语义

`.prj` 不是普通目录清单。它给出：

```text
封面/目录
屏面布置图
二次原理图
背板接线图
元件接线图
屏端子图
```

以及每个类别下的稳定文件顺序。建议把 `.prj` 解析结果提升为 `ProjectProfile`，作为页面类别先验，而非仅用于文件扫描。

## 5. `LdDzbInfo.xml` 可恢复的弱语义

例如第一套项目中包含：

```text
左侧 ZD / CD / KD / 5FD / 3-2QD ...
右侧 1ID / 1-2UD / 1-2ID / 1QD ...
```

该文件提供端子排词表、侧别和长度。可用于：

- 识别 `5FD25` 等完整端子身份；
- 识别 prefix + local number；
- 限制某一页面/区域可能出现的端子排；
- 验证端子页识别完整性；
- 生成弱监督标注。

## 8. 当前 topology 的位置问题

`pipeline.py` 先运行页面 Extractor 和 PairBuilder，`report/artifacts.py` 才调用 `build_wire_topology_frames()`。这使 topology 成为报告证据，而不是连接真值。

现有 topology 具备实际价值：

- endpoint merge；
- orthogonal intersection；
- inline-text gap bridge；
- block-span bridge；
- Union-Find network；
- open endpoint；
- topology shadow report。

但其直接 Union POSSIBLE bridge 的方式不适合成为高可靠主链。

## 9. 仓库自身已经证明“纯拓扑也不够”

`doc/findings.md` 记录：

- 初版 topology shadow 将第一套 37 条残余问题全部判为可恢复；
- 加入文本角色和结构复核后，只有 6/37 属于可信 topology endpoint 问题；
- 其他属于 scoped local number、body-port、semantic local 等结构语义；
- 第二套 6 条候选中 0 条可直接由 topology 接管。

因此，正确结论不是“把所有旧规则换成 Union-Find”，而是：

```text
Topology + Symbol Port + Text Role + Constraint
```
