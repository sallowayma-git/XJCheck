# 交付说明与审查边界

## 审查目标

本次工作的目标不是只给出“建议上 GNN”或“再增加一些规则”，而是回答五个具体问题：

1. 当前 XJCheck 为什么在前两套图上可以不断收敛，但面对第三组多项目语料仍有明显泛化风险；
2. 成熟 ECAD 产品如何组织连接、端子、跨页引用和检查消息；
3. 只有产出后的普通 DWG 时，应如何反向重建一个足以审计的简化电气语义模型；
4. 当前代码应具体移动、拆分、新增和下线哪些模块；
5. 如何建立“运行—发现漏识别—人工复核—沉淀符号/约束/训练样本—回归”的持续闭环。

## 实际输入范围

- 当前仓库：46 个源代码 Python 文件、33 个测试文件，`src + tests` 约 36,407 行 Python；
- `src/dwg_audit/audit/` 约 11,508 行，其中 `wire_topology.py` 1,402 行、`candidates.py` 1,168 行、`rules.py` 1,691 行；
- 当前测试：在解包仓库中以 `PYTHONPATH=.:src pytest -q` 实测 `323 passed`；
- 数据集：27 套项目、502 张 AC1018 DWG；
- 侧车：27 个 `.prj`、27 个 `LdDzbInfo.xml`、27 个 `AirSwitchClassSet.xml`；
- 端子排侧车记录：505 条；
- DWG 内嵌预览：502/502 成功提取。

## 重要限制

本次容器中不存在 ODA/RealDWG/LibreDWG 可执行解析后端。当前仓库的 `dwg_converter.py` 又只自动探测两个 Windows 固定路径。因此，本次无法在容器内对第三组 450 张 DWG 产生真实 `lines/texts/blocks` 实体表。

本包采取了三层补偿：

1. 执行当前扫描器和 sidecar 解析，得到项目、页序、类别和路由风险；
2. 直接从每个 DWG 中提取内嵌 DIB 预览，视觉检查结构域迁移；
3. 提供可在安装 ODA 的目标环境直接运行的基线脚本和失败闭环工具。

所有“已实测”“推断”“需要目标环境复跑”的内容在文档中分开标记。
